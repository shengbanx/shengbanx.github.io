Haxe

初次接触Haxe语言

被“誉为”上帝的语言的Haxe,他是多么的强大。Haxe 项目是Flash开源界的最大贡献。 它的前身是大名鼎鼎的mtasc项目，mtasc 是一款开源的 actionscript 语言（Flash的脚本语言）编译器。2005 年，haxe开始开发。

"One language，everywhere."--HAXE. 什么是Haxe？它有什么特点？Haxe是一门“通用语言”(Universal Language)，使用haxe写的程序，可以编译成不同的其它语言源码，再进行编译。例如用haxe编码，然后可以编译成-JavaScript, Flash, NekoVM, PHP, C++, C# 和Java的源码，再进行编译。这是前所未有的“跨平台”。加上NME模块的支持的话，还支持iOS，Android，WindowsMobile，WebOS。这意味着你的应用可以轻易支持几乎所有流行的智能手机平台。

MVC PureMVC 基础 学习

Flash基础巩固-（2)MVC应用

昨天天看到flash开发者大会在上海举行的微博，看到好多技术大牛在分享他们的对flash的见解。AGAL图像处理、AS4明年的到来、Starling框架、Flash RIA技术、away3d、Stage3d、air对blackberry的支持，等等很多都让我感到自己真的好渺小啊，要学的还有很多！加油！

这两天在了解MVC的应用。感觉mvc是对于一些较大的项目开发比较实用，将程序划分为model、view、control，将有利于对代码的维护、改进，开发起来更加有条理性。model作为模型，创建了程序需要的数据；view视图呈现给用户交互的界面，监听及调用control方法，改变model数据。mvc化能降低程序的耦合度*，使程序模式化。

其中mvc在flash应用中较出名的就是PureMVC了。

PureMVC，就如同它的名字是，它是一个纯粹的小巧的MVC框架，支持多语言。其中支持as2、as3。

PureMVC中还有另外一个单例模式类——Facade，Facade提供了与核心层通信的唯一接口，以简化开发复杂度。

Facade（观察者）统领着MVC三大结构，可以注册、销毁、查询command、proxy、mediator Command（命令）用以操作接受Notification（通知），然后执行相应的动作 Proxy（数据代理）负责提供数据的访问和修改接口，同时随时发布相应的通知告诉外界。 Mediator（视图媒介）负责将界面展现出来，提供用户界面等等，它不但可以接收通知，也可以往外发送通知。 Notification（通知） 这相当于常用的Event，不过功能更强，用以触发Command或者Mediator的处理。而且因为封装过，不会过多依赖开发语言。

Model 与 Proxy

Model保存对Proxy对象的引用，Proxy负责操作数据模型，与远程服务通信存取数据。

这样保证了Model层的可移植性。

View 与 Mediator

View保存对Mediator对象的引用。由Mediator对象来操作具体的视图组件（View Component，例如Flex的DataGrid组件），包括：添加事件*，发送或接收Notification ，直接改变视图组件的状态。

这样做实现了把视图和控制它的逻辑分离开来。
Controller 与 Command

Controller保存所有Command的映射。Command类是无状态的，只在需要时才被创建。
Command可以获取Proxy对象并与之交互，发送Notification，执行其他的Command。经常用于复杂的或系统范围的操作，如应用程序的“启动”和“关闭”。应用程序的业务逻辑应该在这里实现。

Facade 与 Core

Facade类应用单例模式，它负责初始化核心层（Model，View和Controller），并能访问它们的Public方法。

这样，在实际的应用中，你只需继承Facade类创建一个具体的Facade类就可以实现整个MVC模式，并不需要在代码中导入编写Model，View和Controller类。
Proxy、Mediator和Command就可以通过创建的Facade类来相互访问通信。

Observer 与 Notification

PureMVC的通信并不采用Flash的EventDispatcher/Event，因为PureMVC可能运行在没有Flash Event和EventDispatcher类的环境中，它的通信是使用观察者模式以一种松耦合的方式来实现的。你可以不用关心PureMVC的Observer/Notification机制是怎么实现的，它已经在框架内部实现了。你只需要使用一个非常简单的方法从Proxy, Mediator, Command和Facade发送Notification，甚至不需要创建一个Notification实例。

*耦合度：

软件设计中通常用耦合度和内聚度作为衡量模块独立程度的标准。划分摸块的一个准则就是高内聚低耦合。 耦合度是指模块之间联系的紧密程度。 模块间的耦合度是指模块之间的依赖关系，包括控制关系、调用关系、数据传递关系。模块间联系越多，其耦合性越强，同时表明其独立性越差。降低模块间的耦合度能减少模块间的影响，防止对某一模块修改所引起的“牵一发动全身”的水波效应，保证系统设计顺利进行。

Flash MVC 基础

Flash基础巩固-（1)何为MVC？

社团忙得很久没去碰flash开发了，满脑子的想法而没时间去实行，这不能不说也是一种失败。

今天看到一篇文章：“抛弃掉pruemvc吧，flash不适合使用这么死板的东西”，就发现我还不清楚mvc，mvc看过但不知道是什么。于是我就突然有这个一点一滴巩固flash基础的想法。

从现在起20天系统地巩固一下flash的吧，发现一直是系统基础方面比较薄弱，知识了解总是零零散散的，概念也是模模糊糊的，这又谈何开发。正如设计师一样， 不懂得系统的设计规则，又如何去打破它来创新。

 

（以下资料打多来自于百度百科，部分来源于论坛）

mvc为何？

MVC (Model View Controller)。M是指数据模型，V是指用户界面，C则是控制器。模型－视图－控制器（MVC）模式是一个有用的工具箱，它有很多好处，但也有一些缺点。

mvc如何工作？

MVC是一个设计模式，它强制性的使应用程序的输入、处理和输出分开。使用MVC应用程序被分成三个核心部件：模型、视图、控制器。它们各自处理自己的任务。

视图(View)：

代表用户交互界面，MVC设计模式对于视图的处理是限于视图上数据的采集和处理，以及用户的请求，而不包括在视图上的业务流程的处理。

模型(Model)：

1、企业数据和业务规则。在MVC的三个部件中，模型拥有最多的处理任务。被模型返回的数据是中立的，就是说模型与数据格式无关，这样一个模型能为多个视图提供数据。由于应用于模型的代码只需写一次就可以被多个视图重用，所以减少了代码的重复性。

2、业务流程/状态的处理以及业务规则的制定。业务流程的处理过程对其它层来说是黑箱操作，模型接受视图请求的数据，并返回最终的处理结果。业务模型的设计可以说是MVC的核心。

控制(Controller)：

可以理解为从用户接收请求, 将模型与视图匹配在一起，共同完成用户的请求，但是它却不做任何的数据处理。

MVC的优势

开发人员可以只关注整个结构中的其中某一层，可以很容易的用新的实现来替换原有层次的实现，可以降低层与层之间的依赖，有利于标准化，利于各层逻辑的复用。

 

btw，关于框架、模式的区别

有很多程序员往往把框架模式和设计模式混淆，认为MVC是一种设计模式。实际上它们完全是不同的概念。

重用？

构件通常是代码重用，而设计模式是设计重用，框架则介于两者之间，部分代码重用，部分设计重用，有时分析也可重用。在软件生产中有三种级别的重用：内部重用，即在同一应用中能公共使用的抽象块;代码重用，即将通用模块组合成库或工具集，以便在多个应用和领域都能使用；应用框架的重用，即为专用领域提供通用的或现成的基础结构，以获得最高级别的重用性。

框架与设计模式？

框架与设计模式虽然相似，但却有着根本的不同。设计模式是对在某种环境中反复出现的问题以及解决该问题的方案的描述，它比框架更抽象；框架可以用代码表示，也能直接执行或复用，而对模式而言只有实例才能用代码表示;设计模式是比框架更小的元素，一个框架中往往含有一个或多个设计模式，框架总是针对某一特定应用领域，但同一模式却可适用于各种应用。可以说，框架是软件，而设计模式是软件的知识。

框架模式有哪些？MVC、MTV、MVP等等；
框架有哪些？C++语言的QT、MFC、gtk，Java语言的SSH，php语言的 smarty(MVC模式)，python语言的django(MTV模式)等等

设计模式有哪些？工厂模式、适配器模式、策略模式等等

简而言之：框架模式是大智慧，用来对软件设计进行分工；设计模式是小技巧，对具体问题提出解决方案，使得代码复用率和耦合度降低。

工作室给我的

快半年了，说说我对工作室的体会吧。

 

假如我没来维生数工作室，我会后悔的

上个学期的这个时候，我还在担心进不进得了工作室呢？因为我怕技术牛人太多，工作室要求太高，但最终我还是来了，这个名叫“维生数”的工作室。

工作室在这短短一学期里真的给了我很多很多，假如我没来维生数工作室，我会后悔的。这里有我理想中的团队，理想中的学习环境。在这里我很开心，反过来想我却无法对工作室做出什么贡献，我的技术有限，我的能力有限，但接下来我会努力学习技术，我会更加努力地尝试真正成为工作室的一份子，融入工作室，改变工作室，让工作室的明天更好！

工作室让我找到了方向，要坚定地走下去

说一个刚刚发生在我身边的事吧。

刚刚上课坐我旁边的同班同学他突然很颓废很忧愁，两节课下来都这样。放学时我问他怎么了，他说很烦，想死，但这里2楼太矮了，等会回宿舍去7楼楼顶跳下去。我觉得很奇怪，一向开怀的他怎么这样，我说，你怎么了？发生什么事啊。他越走越远（他去不同方向），说，没事，我死之前会跟你说一声的。

刚刚中午他还真敲我宿舍门了，做了个拜拜的手势，就关上了，然后我追出去，只见他跑上楼，我追不及，就发了条让他坚强的短信，电话也不接。然后过了一会他打电话来，说回宿舍睡觉了，说昨晚喝醉了，还没清醒。

我默然，是发生了什么事了吗？随后，我想，可能是吧，人生谁能平坦走过，总有许许多多挫折与困难等待着我们，总要面对，总要经历。但你让我现在去颓废，去寻死，我是不能的了，即使是发生天大的事。因为我还有我的路没走完。

工作室可能给我的不多，因为很多东西需要自己去争取，学AS3代码，学AIR、Flex开发，都要自己花时间去学习。但在其他方面却给了我很多，也让我找到了方向、或者说，是人生的道路。

刚来大学时，我很迷茫，我面对许许多多新鲜的事，我很多计划，我满脑子都在想我要怎样怎样，我以为大学很多时间很自由，但并非如此，活动、社团几乎占用了我大部分的空余时间，我的计划很多都无心去付诸实践。打个比喻，当时摆在我面前的路有很多条，我兜来兜去找不到方向，终点也茫茫。现在，摆在我面前的是一条明确的方向，我要做的就是坚持走下去，坚持做到最好。

因为有和师兄、师姐的交流，潇滨、洁浩、志成、文强等，他们让我更加认识工作室的同时，还让我对大一以后的几年有了大概的想法。还有分享会上大三大四的师兄师姐的分享，也让我对大学有很多感悟，对学技术有更深的认识。

由于我的专业和我现在想做的事完全不沾边，我的专业是材料成型与控制工程，这是个非常工科的专业，总有人问我，为什么要报这个专业，而不报数媒，因为我当初是喜欢机械设计类的专业，这些知识与实践操作时无法通过自学完成的，而flash、代码这些技术可以通过自学学会的。

所以我有时会想，如何处理专业与学技术这两样事，通过师兄师姐的经历，我后来想通了，我要继续坚持走学技术这条路，专业也不能放弃，到时出来工作还是会在IT行业工作。

工作室给我的感觉

我觉得我们的工作室很好，很自由，很温馨。有大一的一群刚起步的伙伴一起学习，一起成长。有大二的一群年龄比较相近的师兄师姐陪同、指导，一起工作、一起玩。有大三的师兄师姐对工作室悉心规划，并分享经验，交流感想。有大四的技术牛的不行的师兄师姐作为榜样，分享他们的作品、对技术的认识，以及实习的体会。还有……

我在工作室认识了很多人，尤其是大一的几个，前端的徐彬、贤杰、盈粲，程序组的庆杰，平面的颖欣、海玲、李镛。我很庆幸认识了他们呢，因为将来他们都有可能成为IT界、设计界的技术牛人。

在工作室，是一个学习技术的好环境，也是一个轻松愉快的环境，学技术不再是孤独的。有人陪着，志成，洁浩，潇滨，海玲，建丽。有时候下午没课，中午一直在工作室呆到晚上11点多才回宿舍的感觉真好。

分享会一直是我期待的，每次都能从师兄师姐身上学到很多东西，不单是技术，还有一些感悟，感想。有次发烧错过了一次分享会我都觉得可惜。话说我也分享了一次，虽然不怎么样。>.<

我觉得，

工作室该有一个自由的学术环境，分享精神，合作精神。

工作室该有一个创新精神或者环境，不再是死板的不知变通。

工作室该有一个优秀的团队，交流、沟通、合作，效率以及成绩！

工作室该有自己的文化，模式。

工作室该有一定的影响力，知名度。

……

我学会了很多，不止技术

有时我感觉很失败。

我没能管理好我的日程、我的时间。这个学期几乎有2个多月的时间被社团忙的不像样，很少来工作室，经历了两个月，回首起来我才发现我的技术却没多少长进。因为我没有学会如何去取舍，如何去抉择，直到后来才学会取舍，我不该浪费太多时间在某些方面，要真正适合自己，别把工作任务都接受，而把自己弄得太辛苦。（有了这样的思想，也就发生了下面的事情）。

我没能很好的与合作的伙伴交流、沟通。

在工作室的初次合作，我、颖欣、贤杰，参加平面动画设计大赛的一个网站作品，颖欣做好网页设计后找到贤杰写代码，贤杰代码实现不了一些效果，让后又找到我做flash，我犹豫了一阵接受了，我当时有工作要完成，比较忙。但接下来的日子，我和颖欣由于初期没沟通好，我做出的flash不是她想要的效果，flash上有点小瑕疵，或者没有哪个效果等等。不停地改动，努力地尝试那代码如何实现。

实在没多少时间，离比赛交作品时间又赶，我时常说：可以了，不用改了，没事的，评委看不出的，一个小比赛而已，要做的话会花我很多的时间去研究，现在效果已经很好了。的却是很好，至少我觉得拿去参加比赛已经没问题了。但她却要那种效果，她说，即使是一像素的差距对设计师来说也是很重要的，她说，如果能做出那种效果就更好了，她说，我不是为了比赛而来的，我是看重这第一次合作……。但我觉得时间上不太允许，而且……，我没有话说了，我说的话、我说的理由都成了我不想工作的借口。好吧，你说的都是对的，我继续改吧TAT。

后来又要做一个网站开头的flash，她在电话一头和我讲了好几遍，我觉得我明白她要做成怎样的了，我那天的明天下午还有考试，还要复习，所以我想尽快完成这个，但花了一个晚上做出的效果，明天中午她跟我说做错了，不是这样，要改已经很麻烦了，浪费时间，况且那个效果也不错了（我觉得）。……接下来太多事情发生了不谈也罢。反正我在这次合作是成了一个让我感觉是“下层”工作者的角色，上头要我做，我不能不做，我在这次合作中有很多不快，我很无奈，我反抗，反抗无效……我承认是沟通这方面没做好，以及团队合作者的相互了解不够。我们双方都有错吧。

我在想，合作就是会遇到这样的问题吗？或者，以后出来工作，面对上司也是这样吗？在汉宽师兄上周的分享中，我才知道，原来他们当时也有过争吵。

我又想，问题不可怕，可怕的是刚刚面对这问题的不知所措，或者直接处不当，导致问题的恶化。是工作室给了我提前经历这些问题的机会，让我能够更加成熟，懂得如何面对这些问题。

在**flash**组

当初报flash是因为不想跟其他人争那些位置，而我有刚好喜欢flash，从小学接触后就开始热爱flash。

随后，进来后发现，flash组大一只招了我一人，平时学习代码时只有文强师兄指导，但我更希望有一个和我一样刚起步的学习伙伴，互相交流，互相竞争、合作。

所以我一直盼望校招的到来，但迟来的校招却没人报flash组，让我些许失望。虽然已让数媒的同学在班群上问了，我还是觉得有些事是要自己去争取的，我亲自去数媒专业的宿舍找有没有有兴趣学flash的同学。后来我通过李镛，问到了他的朋友——海峰，电话一谈，我觉得，就是你了！>.<。

哈哈，现在flash组又招进了两人，海峰和均豪，很开心又有两个伙伴呢。到时做flash的作品时就没那么辛苦要一人做完了。

在flash组，我虽然一个人学，但我还是默默的自学，我觉得我的技术飞涨，从开始很多不会，到现在，也不说都会了，但起码相比还初来工作室时，进步了不止一点半点。

3月有一段时间是很迷茫的，没人交流，找不到要继续学flash的方向。和文强师兄交流也一样什么头绪。我发现了flash组真的如师兄所说，要做的事情比较少，但随着我深入了解逛论坛，发现flash其实能做的不单只网页，还有桌面应用、手机应用等等很多东西，flash还是很强大的。要学Flex、AIR，于是我去图书馆借了相关的书学习。接着阿橙师兄又刚好说要开发活动网手机端的应用，于是我更加深信这个方向是正确的！

暑假，我们flash组就要发力了，虽然文强师兄和我以及海峰、均豪都对AIR还不熟，但我们会努力，争取暑假做出手机端应用来的！

最后

可能工作室现在面临着较多问题，交接传承方面、规章制度方面，以及一些项目工作方面。在这个群邮讨论版面里，我在师兄师姐的话里发现原来工作室现在还很不完善，我们虽然很享受现在的工作室，但工作室不能停留在现有，它要做得更好，它需要我们的付出与努力！不管是大一、大二、大三还是大四毕业的，工作室是大家的。在接下来的日子里，工作室将要大改变，我们要一同努力！！！我相信工作室一定在接下来做得更好！

我敢肯定，我会一辈子记得这个工作室，这个家，以及家里的每一位成员，不管是现在的还是离开的还是未来的。是你们，给了我最好的回忆，是你们，让我深信学技术这条道路是对的，是你们，让我找到了学技术了乐趣！

感谢大家。

开始学习Flex开发！

RIA，也就是Rich Internet Application(富互联网应用程序)。RIA指新一代web应用。

Adobe Flash Builder 4,也就是最新一代的flex版本，今天试着用了一下，发现很强大，那功能。

其实刚开始接触到这方面的概念时，我是想用Flex开发软件用的，因为学习flash as3这么长一段时间了，发现as3除了做一些特效，游戏、网络应用等，很少其他用处，而且逐渐地，有被html5强大的功能取代的趋势。（发现有很多html5做的游戏，不错，不过html5还是无法克服其占用内存太多，运行慢的缺点，做不了大游戏）所以，当知道有flex这东西的时候，我就开始很感兴趣了，开始迷上了RIA这个词，这是一个很棒的词，至少我觉得。我想用As3来开发软件，我从小就想开发自己的软件了，想想，别人都用着自己开发的软件的时候，那是多么得自豪。哈哈，想多了，才刚接触就想这么远的事，有什么用，赶紧学习，学会怎么做吧。

话说，工大在线要改版了，再过两个星期。然后，flash组要讨论一下用FB把工大在线推出一个移动客户端。也就是在安卓等智能机上的应用，不知道可不可行，我才初学，压力山大啊。

as3参考文档 基础 日志

知其然而不知其所以然

我感觉我现在对AS3的学习就是这样，这几天在做个仿切水果的游戏常常遇到问题经过不断尝试解决了后，却很难说出其中的原理。

像昨天忙了一个晚上对一个变量的引用，才搞懂问题所在；以及今天下午尝试了很久才发现添加enterFrame事件侦听器的地方不该在那里；以及对数组的引用。虽然在尝试解决的过程中有了新的认识，对一些代码有更深的了解，但我那感觉还是模棱两可的，因为自己都不敢确定是不是所想的那样。再次遇到同样的问题，却找不到根本的原因，得到最直接的解决办法，而又需要不断地尝试，才会发现另一个解决办法，而还是不知道其中的真正原因。

显然这对于一个代码学习者来说是不好的，虽然这样学会的会印象深刻，但却走了很多弯路，也浪费了很多时间。本身大学空余的时间就不多，我不该在这方面去浪费时间。而且我也感觉我是太过急了，不好好学基础，把基础打劳，就去实践。没完全搞明白化学反应原理，就去搞化学反应，这是同样的道理。

所以，从明天起，我要开始回顾基础，而且要学得深入，明白面向对象编程的方法，搞清楚代码运作的原理。
